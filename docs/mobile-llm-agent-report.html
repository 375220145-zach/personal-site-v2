为什么 North Star 是「周完成任务数」而不是 DAU 或 MAU？

  • — 任务完成数直接衡量 "用户真正靠 PA 在做事"，不是刷一下就跑
  • — 每周粒度平滑了工作日vs周末的波动

激活模型：Aha Moment 的三层信任

品类最大的留存杀手是「用了一次、新鲜感过了、不敢再用了」。激活不是让用户完成一个任务，是建立三层信任：

信任层用户心理PA 设计
第一层：它能行"它真的能帮我发消息？"首次任务成功率 >75%。预设 5 个示例降低决策负担。Agent 先复述再执行
第二层：它不乱来"它不会在我不知道的时候偷看我的东西？"首次任务完成后的隐私摘要弹窗——「本次任务：数据全部在本地，未上传」
第三层：我需要它"手动也可以，但用它更快"技能缓存让重复任务从 60s→8s，用户感知效率提升。推送「你上周在 PA 上完成了 12 个任务，平均每个节省 25 秒」

激活定义: 用户在注册 3 天内完成 ≥3 个任务 且查看了至少一次隐私仪表盘。

留存驱动：4 个习惯环

1. 效率环（日活驱动）

用户每天 1-3 个高频任务（发消息、查信息、设提醒）→ 技能缓存加速 → 感知节省时间 → 第二天继续用

触发: 推送 "早上好，提醒你今天下午 3 点和张三有会，要我帮你准备吗？"

2. 探索环（周活驱动）

PA 自动发现用户 App 使用模式 → 推荐新技能。用户尝试新技能 → 成功 → 技能库 +1

触发: "我注意到你每周用美团至少 3 次，要不要设一个快速复购技能？下次说'再买上次那个'就行。"

3. 信任环（月活驱动）

每周推送隐私报告 → 用户看到 0 次云端请求 → 信任累积 → 愿意让 PA 做更多事

触发: 每周日晚上推送 "你的隐私周报：本周完成 15 个任务，0 次云端请求。2 个新技能缓存到本地。🔒"

4. 社区环（长期留存）

用户创建技能 → 分享到社区 → 获得反馈/Star → 身份认同感 → 继续贡献

触发: "你的'订咖啡'技能被 23 人下载了，排本周 Top 10 🏆"

13. 竞品响应预案 (War Game) 6场景推演 + 概率/影响/应对

核心问题: 12-18 个月的窗口期里，如果巨头做了 X，PA 怎么应对？

场景 1：Google AppFunctions API 公开 GA（概率：中高，时间：2026 Q3-Q4）

触发: Google 在 Android 17 中正式发布 AppFunctions API，允许第三方 App 声明结构化的可操作接口给 Gemini 调用。

影响评估:

  • — Gemini Intelligence 的跨 App 执行能力大幅提升——不需要 GUI 模拟，直接调 API
  • — 首批适配 App：Uber、Spotify、WhatsApp、Google 全家桶
  • — **但对 Pocket Agent 的威胁有限**：AppFunctions 需要开发者主动适配，长尾 App（90%+）不会适配。GUI Agent 永远可以操作任何 App，而 AppFunctions 只覆盖头部

PA 应对:

1. 立即: 不做任何 pivot。PA 的 GUI Agent 路径不受 AppFunctions 影响

2. 3 个月内: 在 PA 中支持 AppFunctions 作为'快速通道'——检测到已适配的 App 时优先用结构化接口，其余仍用 GUI

3. 差异化叙事: "PA 能操作你手机上的所有 App，不只是那些适配了 Google 的"

场景 2：Apple 开放 iOS 侧载（概率：中，时间：2026 Q4-2027 Q1）

触发: 受欧盟 DMA 压力，Apple 在全球范围开放第三方 App Store 或侧载。

影响评估:

  • — PA 的 iOS 版本可以分发——这是重大利好
  • — 但同时会有大量竞品涌入 iOS
  • — Apple 仍会严格限制 Accessibility 类权限

PA 应对:

1. 确认消息后 1 个月内: 发布 iOS Alpha（如果 Android MVP 已验证 PMF）

2. 利用先发: PA 在 Android 上已验证的技能库和社区可以无缝迁移到 iOS

3. iOS 特有策略: 利用 App Intents + Shortcuts（比 Android 的 Accessibility 更结构化）

场景 3：字节跳动宣布豆包开源（概率：低，时间：2026 H2）

触发: 字节受监管或竞争压力，将豆包手机助手核心代码开源。

影响评估:

  • — **这是最危险的场景**——PA 的"开源可审计"差异化被直接削弱
  • — 但豆包即使开源，100+ 权限的架构不会改变（那是商业模型决定的）
  • — 开源不等于可信——用户对字节的隐私信任需要多年修复

PA 应对:

1. 不恐慌: PA 的差异化不是"开源"这一个词，是端侧默认 + 最小权限 + 社区信任的组合

2. 强调信任差异: "豆包即使开源，也是 100+ 权限的云端架构。PA 从第一天就是端侧默认，权限不到豆包的 1/20"

14. 开源治理 License选型 + DCO + 三阶段治理

核心原则: 开源不是把代码扔到 GitHub。社区是 PA 的护城河——治理框架决定这个护城河有多深。

1. License 选型

选项优势劣势PA 适用？
MIT最宽松，商用友好无专利保护，大厂可以拿走闭源❌ 太弱——Google 可以直接集成到 Gemini
Apache 2.0含专利授权，商用友好仍可闭源 fork⚠️ 可接受但不理想
GPL 3.0强制衍生作品开源商用不友好（企业法务会拒绝）❌ 太强——阻碍企业采用
**AGPL 3.0****含网络服务条款——云服务也需要开源**企业法务有一定抵触✅ **推荐** —— 防止 "拿来主义"（大厂 fork 后在云端跑闭源版）
BUSL (Business Source License)商业保护 + 定时转开源不是 OSI 认证⚠️ 可考虑但社区接受度低

推荐: AGPL 3.0 + 贡献者协议（CLA）

理由: PA 的核心价值在端侧，AGPL 对端侧使用几乎没有限制（用户在自己的设备上使用 PA 不触发 AGPL 的网络服务条款）。但如果有人把 PA 包装成云端服务（把 GUI Agent 放服务器上），AGPL 要求他们也开源。这是精准打击——保护 PA 的差异化不被云服务商白嫖，同时不阻碍普通用户和开发者。
